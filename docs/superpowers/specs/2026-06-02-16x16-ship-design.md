# Ship 16×16 (Classic + X-Sudoku) — Design

**Date:** 2026-06-02
**Status:** Approved (design)
**Roadmap link:** Phase 2 item #3 (Multi-size) — the "Remaining to ship 16×16" follow-up.
Builds on the parked `solver-constraint-propagation` branch and ships with it.

## Problem

The engine has been 16×16-capable since the size-generalization work, and the
constraint-propagation work (parked on `solver-constraint-propagation`) made 16×16
generate + grade per-request **sub-second** at the 47% clue floor. But 16×16 is not
exposed to users:

- **Server** rejects `size=16` (`SUPPORTED_SIZES = new Set([6, 9])`, `index.ts:6`).
- **Web** has hard 9×9-max assumptions despite the "size-parametric" claim (which was
  only ever true for 6/9):
  - `Size = 6 | 9` type (`boardState.ts:14`).
  - Notes are a `Uint16Array` — a 16-bit element only holds bits 1..15, so digit 16's
    note bit does not fit (`boardState.ts:19`).
  - Candidate mask is hardcoded `0x3ff` (10 bits, covers 0..9) (`boardState.ts:96`).
  - Digit parse/render is `charCodeAt(0) - 48`, assuming decimal `'1'`–`'9'`; the
    engine emits `A`–`G` for digits 10–16 (`boardState.ts:47`, and render/check sites).
  - `puzzleSize = givens.length === 36 ? 6 : 9` has no 256→16 branch
    (`App.tsx:565` and three sibling sites).
  - NumberPad + keyboard handle 1–9 only; no A–G entry.

## Scope

**This round:** expose 16×16 for **Classic** and **X-Sudoku** end-to-end (server
acceptance + web rendering/input/selector), then merge the whole thing (constraint
propagation + 16×16) to `main` as a single deploy.

**Decisions locked during brainstorming:**

- **Variants:** Classic + X-Sudoku only. Both generate sub-second. Jigsaw (known
  9×9 perf tail of 40s+ on unlucky seeds, worse at 16×16) and Killer (cage UX + grading
  cost at 256 cells) are **deferred** to a later follow-up.
- **Tiers:** decided by measured timing. Benchmark each tier at 16×16 for classic + X
  during implementation; expose whichever stay comfortably sub-second. The resulting
  allowed-tier set is hardcoded per size in the web selector (`TIERS_BY_SIZE`) — no new
  API.
- **Mobile:** 16×16 is offered on all viewports; when picked on a narrow viewport, show
  a gentle "best on a larger screen" hint. No viewport gating of the option itself.
- **Daily:** stays 9×9. The daily/streak path is untouched.
- **Digits 10–16:** rendered as `A`–`G`, matching the engine binaries' output. No
  translation layer — the web speaks the engine's wire format.
- **Notes width:** widen the web notes array `Uint16Array → Uint32Array` universally.
  No behavioral change at 6/9; bit 16 now fits.

**Explicitly out of scope:**

- Jigsaw and Killer at 16×16 (deferred — perf + cage UX).
- 16×16 daily challenge (daily stays 9×9).
- Postgres puzzle pool (#5) — not required for the per-request generate+grade path.
- The pre-existing jigsaw `fill_random` perf tail (separate flagged task).
- Hardening `stillgrid-grade`'s killer `build_variant` against malformed cages
  (separate flagged task; not on the `/api/puzzle` path).

## Approach

**Web — make it fully size-generic for any n ≤ 16** (chosen over special-casing `16`).
`16` becomes "just another `n`": widen the type, derive the candidate mask from `n`,
`Uint32Array` notes, and a single small A–G↔10–16 codec used by parse/render/input.
This removes the 9×9 assumptions properly instead of scattering `if (size === 16)`
branches through render/input/storage.

## Components & changes

**1. Engine** — no code changes; already capable (`--size 16`, A–G output, sub-second
generate+grade at the 47% floor per `bench_16x16_uniqueness_and_generation`). The plan
includes a **verification step**: confirm classic + X-Sudoku generate and grade cleanly
across all tiers at 16×16, and capture the per-tier timing numbers that decide the
exposed-tier set.

**2. Server** (`server/src/index.ts`) — small:
- Add `16` to `SUPPORTED_SIZES`; update the error payload `supported: [6, 9]` → `[6, 9, 16]`.
- Confirm `generate()` forwards `--size 16` and the response echoes `size` (it already
  threads `size`).
- Confirm no per-request spawn timeout that 16×16 would trip (generate ~35 ms +
  grade ~20 ms leaves wide margin, but verify the timeout value).
- Daily route untouched.

**3. Web** —
- `boardState.ts`: `Size = 6 | 9 | 16`; `Uint32Array` notes (update the bit-width
  comment); `n`-derived candidate mask (replace hardcoded `0x3ff`); confirm
  `boxDims(16) → 4×4`; add an A–G codec (`digitToChar` / `charToDigit`) and route all
  parse/render/check through it.
- `App.tsx`: `SizeSelect` gains a 16×16 option (gated to classic/X via the variant the
  user has selected; if a 16×16-unsupported variant is active, 16×16 is disabled with
  the existing disabled-styling); `puzzleSize` derivation handles `givens.length === 256
  → 16` at all four sites; NumberPad renders 1–9 + A–G; keyboard accepts `a`–`g`;
  grid/cell sizing scales for 16; mobile hint when 16×16 is active on a narrow viewport;
  add a 16×16 blurb/hint string ("4×4 boxes / 1–G").
- `TIERS_BY_SIZE`: per-size allowed-tier set, populated from the engine timing
  verification.
- `storage.ts` / `analytics.ts`: already keyed/propped by size; `16` flows through once
  the `Size` type widens. **No `KEY` bump** — `16` is a new value, not a shape change.

## Data flow

Unchanged in shape. `/api/puzzle?variant=classic&size=16&tier=…` → server spawns
`stillgrid-generate --size 16 …` → `stillgrid-grade` (size inferred from input length) →
JSON `{ givens, solution, variant, size: 16, tier }` with digits 10–16 as `A`–`G`. Web
parses A–G via the codec, renders the 16×16 grid, and keys storage/analytics by
`size = 16` (derived from `givens.length`, race-safe — not the selector state).

## Error handling

- Server: `size=16` with an unsupported variant (jigsaw/killer) → `400 unsupported`
  (the variant×size combination is rejected; classic/X only). Decide the exact payload
  during planning (reuse the existing `unsupported size` shape or add a variant note).
- Web: the selector disables 16×16 when a non-classic/X variant is active, so the
  unsupported combination is not reachable from the UI; the server rejection is a
  belt-and-suspenders guard.

## Testing

- **Engine:** per-tier generate+grade verification for classic + X @ 16×16; capture
  timings (drives `TIERS_BY_SIZE`).
- **Web unit tests:** A–G codec round-trip (1–16 ↔ '1'–'9','A'–'G'); `Uint32` notes
  round-trip including bit 16; `puzzleSize` derivation for 256.
- **Server:** `size=16` accepted for classic/X, rejected for jigsaw/killer; `size`
  echoed.
- **Manual:** run the app, screenshot a 16×16 classic + X board on desktop and a narrow
  viewport (confirm the grid, A–G pad, and mobile hint) before shipping.

## Ship

All changes land on top of the parked `solver-constraint-propagation` branch. The whole
branch (constraint propagation + 16×16) then merges to `main` and deploys to prod
together — a single release. Per project convention, run the full CI gates locally
(engine fmt + clippy + test; server + web build + test) before any push to `main`, and
confirm the push explicitly first.
